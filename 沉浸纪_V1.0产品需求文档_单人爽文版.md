# 沉浸纪 V1.0 产品需求文档（单人爽文落地版）

## 0. 文档信息

- 产品名称：沉浸纪（Immersion Chronicles）
- 文档版本：V1.0
- 文档类型：PRD（可评审、可排期、可验收）
- 适用阶段：MVP 到首个正式发布版本
- 产品边界：聚焦单人爽文玩法，保证完整可玩闭环，不追求多人实时复杂联机

---

## 1. 版本目标与范围

### 1.1 版本目标（What）

V1.0 目标是把“单人爽文自由输入”做成稳定、沉浸、可复玩的产品闭环，用户可在一个可控世界中通过自然语言推进剧情，获得至少 20 分钟以上的连续沉浸体验，并支持内容分享与社区交流。

### 1.2 版本成功标准（Why）

1. 用户可以从“选故事”顺利走到“结局+评价+分享”全流程。
2. 自由输入不会导致世界规则崩塌，剧情逻辑保持自洽。
3. 单局时长可稳定达到 20 分钟以上。
4. 用户可创建并发布自定义世界卡，形成基础 UGC 循环。

### 1.3 In Scope（本期纳入）

1. 单人爽文模式落地（核心）
2. 故事大类与子类体系
3. 自定义模式（世界/角色/故事卡）
4. 社区基础能力（评价、心得、玩法、悬赏、组队）
5. 语音反馈（TTS）
6. 文生图（关键节点插图）

### 1.4 Out of Scope（本期不纳入）

1. 多人实时共创战斗或开放剧本杀完整机制
2. 复杂经济系统（交易行、拍卖、跨服排行经济闭环）
3. 全自动高级内容治理体系（先做基础审核与人工兜底）
4. 语音输入（ASR）深度交互闭环（V1.0 仅输出语音播报）

---

## 2. 目标用户与核心场景

### 2.1 目标用户

1. 核心用户：20-35 岁网文/短剧/剧情游戏爱好者，偏好“高爽点、可代入、可反转”。
2. 潜在用户：轻度 RPG 用户与 AI 聊天产品用户，愿意尝试沉浸式角色扮演。
3. 创作者用户：愿意设计世界观和剧情模板，并希望被他人体验与反馈。

### 2.2 核心使用场景

1. 碎片与整段娱乐：地铁、午休、晚间 20-40 分钟沉浸体验。
2. 情绪宣泄：通过“反击、逆袭、成长”获得即时情绪回报。
3. 创作表达：构建自定义世界并邀请他人试玩。
4. 社区社交：发布战报、交流玩法、悬赏挑战、发起组队。

---

## 3. 产品定位与价值主张

### 3.1 产品定位

“可控自由输入”的单人爽文叙事游戏：用户自由表达，系统在规则内高质量响应，兼顾爽感和逻辑。

### 3.2 核心价值主张

1. 在其他互动产品里用户“选主角”，在沉浸纪用户“成为主角”。
2. 非固定选项叙事，用户语言直接影响剧情方向。
3. 每局保证成长反馈与高光时刻，形成可感知爽点。
4. 创作卡 + 社区交流形成内容复玩与传播。

---

## 4. 版本功能清单（完整）

### 4.1 单人爽文模式（P0）

#### 4.1.1 故事大类

1. 古风
2. 系统
3. 末日
4. 现代
5. 自定义

#### 4.1.2 子类分区（示例）

- 古风：宗门修行、王朝权谋、江湖恩怨
- 系统：签到变强、任务升级、属性养成
- 末日：求生据点、异能进化、资源争夺
- 现代：职场逆袭、都市悬疑、情感成长
- 自定义：用户自行定义世界规则与角色关系

#### 4.1.3 标准玩法流程

1. 进入游戏大厅，选择大类与子类。
2. 查看故事简介、标签、预估时长、难度。
3. 开局后 AI 叙事引导，用户自由输入行动/对话/策略/情绪。
4. 系统进行规则校验与状态更新，生成剧情反馈。
5. 持续推进至结局（单局不少于 20 分钟）。
6. 结算并引导评价、分享、进入社区。

### 4.2 自定义模式（P0）

#### 4.2.1 创作对象

1. 世界卡：背景、规则、时代、禁忌、资源体系
2. 角色卡：身份、动机、关系、能力、台词风格
3. 故事卡：起始冲突、关键锚点、结局方向

#### 4.2.2 发布与消费

1. 用户可保存草稿。
2. 用户可发布为可玩内容（本人可玩/公开可玩）。
3. 其他玩家可进入体验并评分、评论。

### 4.3 社区系统（P0）

1. 故事评价：星级与文字评价。
2. 沟通心得：体验记录、剧情感想。
3. 玩法交流：策略讨论、流派分享。
4. 任务悬赏：发起挑战任务、征集攻略。
5. 组队入口：为后续多人模式预埋（V1.0 先提供组织能力，不做实时多人战斗）。

### 4.4 语音反馈（P0.5）

1. TTS 播报开关（默认关闭，可手动开启）。
2. 播报内容：优先“行动结果 + 情感体验”。
3. 支持 2-3 种音色与倍速。
4. 不阻塞主流程，异步生成。

### 4.5 文生图（P0.5）

1. 关键节点插图：开场、高潮、结局（每局最多 3 张）。
2. 风格随故事大类自动切换。
3. 允许有限重绘（防滥用）。
4. 图片审核与违规兜底图。

## 5. 信息架构与页面规划

### 5.1 一级导航

1. 游戏大厅
2. 创作工坊
3. 社区
4. 我的（进度、收藏、历史）

### 5.2 页面明细

#### 5.2.1 游戏大厅

- 大类卡片、子类筛选、推荐榜、最新/热门。

#### 5.2.2 故事详情页

- 背景介绍、标签、预估时长、难度、作者、评分、开局按钮。

#### 5.2.3 游戏进行页（核心）

- 剧情流（对话/叙述）
- 自由输入框（含智能提示）
- 状态栏（等级/技能/关系/资源）
- 语音播放控件
- 插图展示位（关键节点）

#### 5.2.4 结算页

- 关键事件回顾
- 能力与关系变化
- 结局标签
- 评分/评论
- 分享到社区

#### 5.2.5 创作工坊

- 世界卡编辑器
- 角色卡编辑器
- 故事卡编辑器
- 发布设置

#### 5.2.6 社区页

- 评价区、心得区、玩法区、悬赏区、组队区

## 6. 核心交互与叙事机制

### 6.1 自由输入类型

1. 行动输入（攻击、探索、使用道具等）
2. 对话输入（威慑、协商、试探等）
3. 策略输入（示弱、诱敌、布局等）
4. 情绪输入（紧张、愤怒、悲伤、坚定）

### 6.2 单回合反馈结构

1. 行动结果（主反馈）
2. 情感体验（代入反馈）
3. 成长反馈（数值/关系/物品）
4. 新挑战（下一轮钩子）

### 6.3 20 分钟时长保障机制

1. 锚点节奏：开场冲突 -> 小高潮 -> 转折 -> 大高潮 -> 结局。
2. 结果预算：每轮 1 主结果 + 最多 1 副作用 + 最多 1 钩子。
3. 结局门槛：至少触发 N 个关键锚点后才能结算。
4. 动态调速：用户输入太快时增加事件层次，输入较慢时加强引导和短反馈。

---

## 7. 防崩塌协议（V1.0 强制）

### 7.1 总原则

用户表达可以自由，系统执行必须受限：先规则校验，再状态提交，后文案渲染。

### 7.2 五类硬校验

1. 能力边界校验（不可越级开挂）
2. 场景规则校验（禁区、规则、职业限制）
3. 时间连续性校验（不可瞬移跨时段）
4. NPC 一致性校验（动机与立场不可跳变）
5. 任务节奏校验（不可一轮推完主线）

### 7.3 越界输入处理

1. 软改写：降级为合法近似动作。
2. 拆分执行：多意图分轮处理。
3. 代价化执行：允许高光但附带代价。
4. 单句澄清：无法判定时触发。

### 7.4 状态先行机制

1. 先生成 `state_delta`。
2. 通过一致性检查后才允许文案生成。
3. 文案不得新增 `state_delta` 之外事实。

---

## 8. AI 能力设计

### 8.1 核心 AI 模块

1. 意图解析器：将用户自然语言转结构化意图。
2. 世界规则引擎：执行硬规则校验与降级策略。
3. 叙事生成器：在合法状态上生成高爽点文案。
4. 节奏调谐器：保证 20 分钟曲线与高潮分布。
5. 安全守卫：文本/语音/图片内容审核。

### 8.2 多模态输出策略

1. 文本为主，语音与图片为增强层。
2. 文本先返回，语音/图片异步补充。
3. 语音与图片均受状态机约束与审核流程约束。

---

## 9. 数据模型（V1.0 最小集合）

### 9.1 游戏数据

1. 故事大类、子类、标签、难度、时长预估。
2. 世界状态：场景、时间、规则、事件标记。
3. 玩家状态：等级、技能、资源、关系、成就。
4. 锚点状态：已触发锚点、剩余锚点、结局条件。

### 9.2 创作数据

1. 世界卡结构化字段。
2. 角色卡结构化字段。
3. 故事卡结构化字段。
4. 发布与版本管理信息。

### 9.3 社区数据

1. 评分、评论、点赞、收藏。
2. 悬赏任务与参与记录。
3. 组队贴与报名记录。

---

## 10. 运营与内容策略

### 10.1 首发内容策略

1. 五大类同时上线，但每类先投放 3-5 个高质量模板。
2. 优先打磨高留存题材：古风修行、现代逆袭、末日求生。
3. 每周上新子类事件包，强化复玩。

### 10.2 社区冷启动策略

1. 官方发布“高光输入示例帖”引导表达。
2. 发起“本周最爽反击”悬赏活动。
3. 创作工坊优秀卡片进入推荐位。

---

## 11. 指标体系（可验收）

### 11.1 核心指标

1. 中位单局时长 >= 20 分钟。
2. 完整通关率（从开局到结算）达到目标阈值。
3. 叙事一致性错误率（规则冲突/角色失忆）低于目标阈值。

### 11.2 体验指标

1. 用户满意度评分（自由输入特性）
2. 每局有效输入轮次
3. 爽点命中率（用户标注“高光”回合占比）

### 11.3 UGC 与社区指标

1. 世界卡发布率
2. 世界卡被试玩率
3. 人均社区互动（评论/点赞/发帖）
4. 任务悬赏参与率与完成率

### 11.4 多模态指标

1. TTS 启用率与播放完成率
2. 文生图触发率与分享率
3. 多模态开启后对会话时长与完局率的影响

---

## 12. 技术与性能要求

### 12.1 响应要求

1. 主文本响应优先（目标 <3 秒，峰值可降级）。
2. 语音与图片异步返回，不阻塞输入。
3. 失败兜底：语音失败回落文本，图片失败回落默认场景图。

### 12.2 成本控制

1. 分层模型策略：简单输入轻模型，复杂输入重模型。
2. 缓存复用：高频场景模板与风格模板缓存。
3. 文生图配额：按局/按日限额。

### 12.3 安全与合规

1. 文本、语音、图片统一走审核。
2. 敏感输出触发重写或拦截。
3. 社区内容支持举报、下架、申诉流程。

---

## 13. 里程碑与排期建议

### Phase A（0-6 周）核心闭环

1. 单人主流程（选故事 -> 游玩 -> 结算）
2. 五大类首批模板
3. 防崩塌协议接入
4. 基础社区（评价、心得）

### Phase B（7-10 周）体验增强

1. 自定义模式（世界/角色/故事卡）
2. 社区扩展（玩法、悬赏、组队入口）
3. TTS 上线

### Phase C（11-14 周）多模态与优化

1. 关键节点文生图
2. A/B 测试与时长优化
3. 上线前稳定性与审核能力补强

---

## 14. 验收清单（上线门槛）

1. 五大类均可进入并可完整通关。
2. 单局中位时长达到 20 分钟目标。
3. 防崩塌规则生效，关键一致性错误可控。
4. 自定义卡可创建、保存、发布、试玩。
5. 社区五类基础功能可用。
6. TTS 可用且不阻塞流程。
7. 文生图在关键节点可触发且有审核兜底。

---

## 15. 风险与应对

1. 叙事失控风险：用状态先行与硬规则拦截。
2. 成本超支风险：分层模型 + 配额 + 缓存。
3. 时长不足风险：锚点节奏和结局门槛动态调优。
4. 社区内容风险：审核 + 举报 + 分级处罚。
5. 多模态拖慢风险：异步生成与失败回落。

---

## 16. 版本结论

V1.0 的正确目标不是“无限自由”，而是“可控自由”。在单人爽文主赛道里，通过规则约束、节奏设计与多模态增强，沉浸纪可以先建立稳定的可玩闭环，再向多人共创与更深生态扩展。
